What to notice. Everything runs locally. No databases, no cloud services, no auth provider. The proxy is what avoids CORS during development; production would terminate at a reverse proxy (nginx, ALB, etc.) instead.

Frontend layering

Angular standalone components consume the NgRx store through per-feature facades. Signals appear at the leaf via toSignal(). Two HTTP interceptors sit between the services and the network.

What to notice. Components never import from Store directly — they go through a facade. The facade exposes Observables (items$, loading$); components decide whether to consume them as observables or bridge them to signals via toSignal().

Exception: the audit log bypasses NgRx entirely — it’s read-only, single-view, and a full feature slice would be five files of boilerplate for one GET endpoint.

Section 04 · Behaviour

Request lifecycle end-to-end

A single POST /api/employees request, traced through every box that touches it. Same pattern applies to PUT, PATCH, DELETE. 

sequenceDiagram
    participant User as Admin
    participant View as Form
    participant Facade as Facade
    participant Effect as create effect
    participant API as ApiService
    participant CidI as cid interceptor
    participant Express as Express
    participant Ctrl as Controller
    participant Val as Validator
    participant Svc as Service
    participant Repo as Repository
    participant Audit as AuditService
    participant Store as data/store

    User->>View: submit form
    View->>Facade: facade.create(payload)
    Facade->>Effect: dispatch action
    Effect->>API: api.create(payload)
    API->>CidI: HTTP POST
    CidI->>Express: + X-Correlation-Id
    Express->>Ctrl: route handler
    Ctrl->>Val: validateCreate(body)
    Val-->>Ctrl: errors[] (empty)
    Ctrl->>Svc: service.create(payload, ctx)
    Svc->>Repo: insert(employee)
    Repo->>Store: push
    Svc->>Audit: recordEmployeeCreated
    Audit->>Store: append entry
    Svc-->>Ctrl: employee
    Ctrl-->>API: 201 + JSON
    API-->>Effect: createSuccess
    Effect-->>Facade: reducer updates items[]
    Facade-->>View: signal emits
    View-->>User: redirect + toast

What this proves. The single source of truth is the store. The view never holds a copy of the employee — it reads through the facade, which reads the store. Swap the in-memory store for Postgres tomorrow and only the repository changes.

Section 07 · User flow

Deleting an employee with cascade soft-close

The most complex single action in the system. One DELETE produces multiple audit entries: one DELETE for the employee and one CASCADE_CLOSE per OPEN account they owned. 

sequenceDiagram
    actor Admin
    participant Detail as Detail page
    participant Dialog as Confirm dialog
    participant Facade as EmployeeFacade
    participant API as DELETE /employees/:id
    participant EmpSvc as EmployeeService
    participant EmpRepo as EmployeeRepo
    participant AcctRepo as AccountRepo
    participant Audit as AuditService

    Admin->>Detail: click "Delete"
    Detail->>Dialog: confirmOpen.set(true)
focus moves to Cancel
    Admin->>Dialog: click "Delete"
    Dialog->>Detail: (confirmed) emit
    Detail->>Facade: facade.delete(id)
    Facade->>API: DELETE request

    API->>EmpSvc: remove(id, ctx)
    EmpSvc->>EmpRepo: findById(id)
    EmpRepo-->>EmpSvc: employee
    EmpSvc->>AcctRepo: findByEmployeeId
    AcctRepo-->>EmpSvc: OPEN accounts[]
    EmpSvc->>EmpRepo: deleteById
    EmpSvc->>AcctRepo: updateAllByEmployeeId
(status: CLOSED)
    EmpSvc->>Audit: recordEmployeeDeleted
    loop For each OPEN account
        EmpSvc->>Audit: recordAccountCascadeClosed
    end
    EmpSvc-->>API: 204 No Content
    API-->>Facade: deleteSuccess
    Facade->>Detail: navigate back
focus restored
    Detail->>Admin: toast "Employee deleted"
The audit trail outlives the deleted row. GET /api/employees/:id/audit still resolves even though the row is gone — audit entries are append-only and never depend on a live FK target.
Section 08 · User flow

Closing & reopening an account

Status-flip flows demonstrate a non-trivial audit-narrative decision: a CLOSE through the named endpoint produces a CLOSE audit action, but the same status change via a generic PATCH produces an UPDATE with a diff. The service layer makes the call. 

flowchart TB
    Start([User clicks Close])
    Confirm{Confirm dialog}
    Cancel([User cancels])
    Submit[User clicks Close account]
    PatchOrDelete[/DELETE /api/accounts/:id/]
    Service[AccountService.close]
    UpdateRepo[Repository: status = CLOSED]
    AuditClose[AuditService.recordAccountClosed
action: CLOSE]
    Done[Toast + CLOSED badge
Edit/Close → Reopen]

    Reopen([User clicks Reopen])
    Patch[/PATCH /api/accounts/:id
status: OPEN/]
    PatchSvc[AccountService.patch]
    StatusOnly{Only a status
flip to OPEN?}
    AuditReopen[recordAccountReopened
action: REOPEN]
    AuditUpdate[recordAccountUpdated
action: UPDATE]
    DoneReopen[Toast + OPEN badge]

    Start --> Confirm
    Confirm -->|cancel| Cancel
    Confirm -->|confirm| Submit
    Submit --> PatchOrDelete
    PatchOrDelete --> Service
    Service --> UpdateRepo
    Service --> AuditClose
    AuditClose --> Done

    Reopen --> Patch
    Patch --> PatchSvc
    PatchSvc --> StatusOnly
    StatusOnly -->|yes| AuditReopen
    StatusOnly -->|no| AuditUpdate
    AuditReopen --> DoneReopen
    AuditUpdate --> DoneReopen

    style AuditClose fill:#EEF2FF,stroke:#1f4ed8
    style AuditReopen fill:#E8F5E8,stroke:#008A00
    style AuditUpdate fill:#FFF6DA,stroke:#B45309

The interesting branch. AccountService.patch() checks whether the incoming patch is only a status change. Multi-field PATCHes always produce UPDATE, never the named narrative. This decision lives in the service so the repository stays a dumb append target.

The boundary. NgRx owns global state. The facade is the public surface. Signals live only inside the component layer. Components dispatch into the store via the facade; data flows out of the store back via signals.
Section 10 · Cross-cutting

Correlation-id end-to-end

Every request carries an X-Correlation-Id header so client logs, server logs, and audit entries can be matched up. 

sequenceDiagram
    participant Client as Angular client
    participant CidI as cid interceptor
    participant Express as Express middleware
    participant Logger as morgan logger
    participant Ctrl as Controller
    participant Svc as Service
    participant Audit as AuditService
    participant Logs as Server stdout

    Client->>CidI: any HTTP request
    CidI->>CidI: generate uuid
(or crypto.randomUUID)
    CidI->>Express: + X-Correlation-Id: abc-123
    Express->>Express: echo header back
attach req.correlationId
    Express->>Logger: req with cid
    Logger->>Logs: "POST 201 12ms - cid=abc-123"
    Express->>Ctrl: route handler
    Ctrl->>Svc: service.method(
payload, { correlationId, actor })
    Svc->>Audit: record* entries include cid

Result. A user reports "the delete button didn't work" → pull the cid from the browser console, grep server logs for that exact cid, see every request/response in that conversation, find the matching audit entry.

Section 11 · Cross-cutting

Audit recording pipeline

Every write in the service layer flows into one of six narrative paths. No-op UPDATEs are dropped so the trail doesn’t fill with noise. 

flowchart TB
    Write([Any write in a service])
    SnapBefore[Snapshot via findById]
    DoWrite[repository write]
    SnapAfter[Snapshot after]
    Decide{Action type?}

    Create[recordEmployeeCreated
snapshot]
    Delete[recordEmployeeDeleted
snapshot pre-delete]
    Update[recordEmployeeUpdated
diff before vs after]
    Close[recordAccountClosed]
    Reopen[recordAccountReopened]
    Cascade[recordAccountCascadeClosed
+ reason]

    NoChange{Diff result empty?}
    Skip([Return null — no entry])

    Compose[Compose entry:
entryId · employeeId
resource · action
actor · correlationId
timestamp]
    Append[AuditRepository.append]
    Done([Entry persists forever])

    Write --> SnapBefore --> DoWrite --> SnapAfter --> Decide
    Decide -->|create| Create
    Decide -->|delete| Delete
    Decide -->|update| Update
    Decide -->|close| Close
    Decide -->|reopen| Reopen
    Decide -->|cascade| Cascade
    Update --> NoChange
    NoChange -->|yes| Skip
    NoChange -->|no| Compose
    Create --> Compose
    Delete --> Compose
    Close --> Compose
    Reopen --> Compose
    Cascade --> Compose
    Compose --> Append --> Done

    style Skip fill:#FBE7EB,stroke:#C7102E
    style Done fill:#E8F5E8,stroke:#008A00
The rule. No-op UPDATEs don’t pollute the trail. Every other action always produces an entry. This rule lives in the service, not the controller, so any path that calls the service inherits it.
Section 12 · Structure

Data model — entity relationships

Three entities. The audit trail is a child of both Employee and Account, but its FKs are nominal — they survive deletion of the parent row by design. 

erDiagram
    EMPLOYEE ||--o{ ACCOUNT : owns
    EMPLOYEE ||--o{ AUDIT_ENTRY : has_trail_of
    ACCOUNT ||--o{ AUDIT_ENTRY : has_trail_of

    EMPLOYEE {
        string employeeId PK "UUID"
        string firstName "2-60 chars"
        string lastName "2-60 chars"
        string email "unique"
        string role "ADMIN|MANAGER|SUPPORT"
        string status "ACTIVE|INACTIVE"
        string createdAt
        string updatedAt
    }

    ACCOUNT {
        string accountId PK "UUID"
        string employeeId FK "→ Employee"
        string accountNumber "8-19 digits, unique"
        string accountType "CHECKING|SAVINGS"
        string currency "CAD|USD"
        number balance "0 to MAX_BALANCE"
        string status "OPEN|CLOSED"
        string createdAt
        string updatedAt
    }

    AUDIT_ENTRY {
        string entryId PK "UUID"
        string employeeId FK "outlives row"
        string resource "Employee|Account"
        string resourceId
        string action "6 values"
        string actor "currently admin"
        string correlationId
        string timestamp
        json snapshot "CREATE / DELETE"
        json changes "UPDATE diff"
        string accountNumber "for Account"
        string reason "for CASCADE_CLOSE"
    }

Key constraint not visible above. AUDIT_ENTRY.employeeId is an FK in spirit only. When an employee is hard-deleted, the audit entries persist by design — the trail outlives the row. The FK column means “this entry belongs to the audit trail named by this employee id”, not “this entry has a live FK target”.

Closing

What to take from this

Every diagram in this document describes a real piece of the codebase you can open and point at. The system architecture matches app.js + app.config.ts; the layered backend matches the folder layout under server/; the request lifecycle matches what you see in the Network tab when you click a button.

None of this is aspirational — it’s all in the repo. Drawing it on a whiteboard during the interview is a 60-second exercise because it was drawn here first.

“The interviewer doesn’t care how many lines of code I wrote, what frameworks I used, or whether every test passes. They care whether I can stand at a whiteboard, sketch the system, name the alternatives I rejected, point at three specific things I’d change today, and ask better questions than they expected. The code is just the prop. The thinking is the show.”
