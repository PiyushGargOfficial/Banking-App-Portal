## 1. Pre-flight: what must be running

Cypress drives a **real browser against the real app**, so both servers must be
up first. The e2e `baseUrl` is the Angular dev server, which proxies `/api/*` to
the Express mock ([cypress.config.ts](../client/cypress.config.ts)).

```
┌─────────────┐    visits http://localhost:4200     ┌──────────────────────┐
│   Cypress   │ ──────────────────────────────────► │  Angular dev server  │
│  (browser)  │                                     │        :4200         │
└─────────────┘                                     └──────────┬───────────┘
       ▲   clicks, types, asserts on the real DOM               │ proxy /api/*
       │                                                        ▼
       │                                            ┌──────────────────────┐
       └──────────── sees responses ─────────────── │  Express mock API    │
                                                     │   :3000 (in-memory)  │
                                                     └──────────────────────┘
```

**One command starts both** (run in the repo root, leave it running in its own terminal):

```bash
npm start          # concurrently boots Express :3000 + Angular :4200
```

Sanity check: open <http://localhost:4200> — you should see the seeded employee
list (33 employees). Now Cypress has something to drive.

## 2. The two ways to run (and which to demo)

| Command (from repo root) | What it does | Use in an interview? |
|---|---|---|
| `npm --prefix client run e2e:open` | Opens the **interactive Cypress runner** (`cypress open`) — a browser you watch step through the UI | ✅ **Yes — this is the demo.** Visual, narratable |
| `npm run e2e` | **Headless** (`cypress run`) — no window, prints pass/fail in the terminal | Mention it as "what CI runs" |
| `npm run e2e:api` | API-only suite via `start-server-and-test` (boots the server itself, no browser) | Mention as the contract-level tests |

> Both browser commands assume `npm start` is already running. The API suite
> (`e2e:api`) boots its own server, so it needs nothing running first.

**The interview move:** demo with `e2e:open` (let them *watch* a spec walk the UI),
then say *"the same specs run headless in CI via `npm run e2e`."*

## 4. What to explain during the end-to-end flow

Open `employee-flow.cy.ts` and walk the **create + delete** test — it's the
richest one. Here's the spec, then the sequence, then the talking points.

```ts
it('creates a new employee, then deletes it', () => {
  // unique email so re-runs don't collide with earlier data
  const uniqueEmail = `cy.user.${Date.now()}@bankadmin.io`;

  cy.dataCy('new-employee').click();
  cy.url().should('include', '/employees/new');

  cy.dataCy('firstName').type('Cypress');
  cy.dataCy('lastName').type('User');
  cy.dataCy('email').type(uniqueEmail).blur();
  // wait for the async unique-email validator to settle
  cy.dataCy('email').should('not.have.class', 'ng-pending');
  cy.dataCy('role').select('SUPPORT');
  cy.dataCy('status').select('ACTIVE');

  cy.dataCy('submit').click();

  // a success effect navigates to the new detail page
  cy.url().should('match', /\/employees\/[0-9a-f-]+$/);
  cy.dataCy('employee-summary').should('contain', 'SUPPORT');

  cy.dataCy('delete-employee').click();
  cy.dataCy('confirm-ok').click();

  cy.url().should('match', /\/employees\b/);
  cy.dataCy('employee-table').should('not.contain', uniqueEmail);
});
```

### The end-to-end sequence (what's really happening)

```
Cypress          Angular (NgRx)            Express (:3000)         in-memory store
  │  click "new"     │                          │                        │
  │ ───────────────► │ route → /employees/new   │                        │
  │  type + blur     │                          │                        │
  │ ───────────────► │ async validator ───────► │ GET /email-available ─►│ (unique? yes)
  │  click submit    │ dispatch create ───────► │ POST /employees ──────►│ insert + audit
  │                  │ effect: navigate + toast │                        │
  │  assert URL /:id │ ◄─── 201 + new employee  │                        │
  │  click delete    │                          │                        │
  │  confirm-ok      │ dispatch delete ───────► │ DELETE /employees/:id ►│ remove + cascade
  │  assert gone     │ ◄─── back to list        │                        │
```

### Talking points (say these as it runs)

1. **`data-cy` selectors, not CSS.** Every step uses `cy.dataCy('…')`, a custom
   command ([commands.ts](../client/cypress/support/commands.ts)) that resolves
   `[data-cy="…"]`. *"Tests target stable test hooks, so a CSS refactor or a
   restyle never breaks them."*

   ```ts
   Cypress.Commands.add('dataCy', (name) => cy.get(`[data-cy="${name}"]`));
   ```

2. **Unique email per run.** `cy.user.${Date.now()}@…` — *"the backend is
   in-memory and persists within a session, so I generate a unique value instead
   of depending on a clean database. The tests are order- and run-independent."*

3. **Waiting on the async validator, not on a timer.** `should('not.have.class',
   'ng-pending')` — *"I wait for Angular's pending state to clear, so the test
   syncs with the real async unique-email check instead of an arbitrary
   `cy.wait(500)`. No flakiness."*

4. **Asserting the effect-driven navigation.** `cy.url().should('match', …)` —
   *"On success an NgRx effect navigates to the new detail page; the test asserts
   that, which proves the whole action→effect→router chain fired."*

5. **The confirm dialog is exercised, not bypassed.** `delete-employee` →
   `confirm-ok` — *"the delete goes through the real accessible confirm dialog."*

6. **It proves the full stack.** *"One test touches the component, NgRx, the HTTP
   interceptor, Express routing, the service's create + cascade-delete, and the
   in-memory store — end to end, in a real browser."*

## 5. The four specs — what each proves

| Spec | Flow it covers | The point to make |
|---|---|---|
| `employee-flow.cy.ts` | List, search, create + delete, required-field validation | The core CRUD happy path + form validation |
| `account-flow.cy.ts` | Account CRUD on the detail page; balance/format validation; close via confirm; reopen | Parent→child (detail→accounts), soft-close, masked numbers |
| `audit-log-flow.cy.ts` | After each action, the matching entry appears at the **top** of the audit log with the right label, masked account number, actor, cid | The audit trail works end to end |
| `employee-filters.cy.ts` | search/role/status/hasAccounts narrow + compose with AND; reset clears all; any change → page 1 | Filtering semantics + pagination reset |

*"Coverage is on the four highest-traffic user journeys — not every permutation,
but the flows a regression would actually hurt."*

## 7. Gotchas to mention before they ask

Naming these unprompted reads as senior:

- **Both servers must be up.** `e2e` / `e2e:open` assume `npm start` is running
  (baseUrl `:4200`). Only `e2e:api` boots its own server.
- **In-memory store persists within a session.** That's *why* the create test
  uses a `Date.now()` email — so the suite is re-runnable without a reset
  endpoint. Restarting the server reseeds.
- **No fixed `cy.wait(ms)`.** Waits are tied to real app state (`ng-pending`, URL
  changes), which is what keeps the suite non-flaky.
- **Two configs.** `cypress.config.ts` (browser e2e, baseUrl :4200) and
  `cypress.config.api.ts` (API-level `cy.request` tests, no browser).

## 9. Troubleshooting: Cypress binary won't install (Windows)

A real issue hit on this project — documented so the next person doesn't lose an
hour to it.

### Symptoms

- `cypress install` (or the postinstall during `npm install`) **hangs forever at
  `Unzipping Cypress  0% 0s`** — or silently finishes having extracted only **one**
  file (`browser_v8_context_snapshot.bin`).
- `cypress verify` reports **`Cypress binary version: not installed`** even though
  the npm package is present.
- Bonus gotcha: typing `cypress install` directly gives *"'cypress' is not
  recognized…"* — because **Cypress is a local devDependency, not a global
  command.** Always invoke it as `npm --prefix client exec -- cypress <args>` (from
  the repo root) or `npx cypress <args>` (from inside `client/`).

### Root cause

**Cypress's own bundled unzipper hangs on some Windows setups.** It is *not*
antivirus — verified by extracting the same zip with PowerShell's `Expand-Archive`,
which unpacked all **21,953** files cleanly (AV would have blocked that too). So
the fix is to bypass Cypress's unzipper: download the zip and extract it with a
different tool, straight into Cypress's cache.

### The fix (PowerShell)

Run these from the repo root. `$ver` must match the Cypress version in
`client/package.json` (here, `13.17.0`):

```powershell
$ver   = "13.17.0"
$zip   = "$env:TEMP\cypress-$ver.zip"
$dest  = "$env:TEMP\cypress-extract"
$cache = "$env:LOCALAPPDATA\Cypress\Cache\$ver\Cypress"

# 1. clear any partial/corrupt cache from the failed unzip
npm --prefix client exec -- cypress cache clear

# 2. download the binary zip for this platform (win32-x64)
$ProgressPreference = 'SilentlyContinue'
Invoke-WebRequest "https://download.cypress.io/desktop/$ver?platform=win32&arch=x64" `
  -OutFile $zip -UseBasicParsing

# 3. extract with PowerShell's unzip (NOT Cypress's) — the actual fix
Expand-Archive -Path $zip -DestinationPath $dest -Force

# 4. place it exactly where Cypress looks for it: <cache>\<ver>\Cypress\Cypress.exe
if (Test-Path $cache) { Remove-Item -Recurse -Force $cache }
New-Item -ItemType Directory -Force -Path (Split-Path $cache) | Out-Null
Move-Item "$dest\Cypress" $cache        # instant rename — both paths are on C:

# 5. confirm, then clean up
npm --prefix client exec -- cypress verify
Remove-Item $zip -Force; Remove-Item $dest -Recurse -Force -ErrorAction SilentlyContinue
```

A healthy result has **~21,953 files** under `…\Cache\13.17.0\Cypress\` with
`Cypress.exe` present — not the single file the failed unzip leaves behind.

### Notes

- **Run `cypress verify` in an interactive desktop terminal**, not a headless/CI
  shell. The verify step launches Electron's GUI smoke test; with no display it
  fails with `bad option: --ping=NNN` even when the binary is perfectly fine.
- **Re-running `npm install` / `cypress install` can re-trigger the hang** (it
  invokes Cypress's unzipper again). If it does, just repeat the steps above — the
  download is cached state-independent.
- Adding a Windows Defender exclusion for `…\Cypress\Cache` does **not** fix this
  (AV isn't the cause) — skip it.
- The `bad option: --ping` GUI failure on a *real* desktop usually means a
  graphics issue; allowing software rendering (`ELECTRON_DISABLE_GPU=1` or
  Cypress's `--disable-gpu`) is the next thing to try there.
